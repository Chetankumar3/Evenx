================================================================================
EVENT BOOKING SYSTEM: ARCHITECTURE & IMPLEMENTATION PLAN
================================================================================

[FUTURE SCOPES]
- Distributed Unlocker with Leader Consensus.
- Dedicated Message Broker to handle a high number of Subscribers.
- Dedicated BoSS replicas for segmented high-demand events (e.g., peak concurrency hours during announcements) to handle massive view/select traffic.
- Advanced Cache Failure Handling (Rehydration/Lazy Loading): If the Redis bitmap is destroyed or returns null, pause requests, fetch actual bookings from the persistent DB, and reconstruct the bitmap string before proceeding.
- Robust Database Transaction Recovery: Handling edge cases where payment succeeds but the DB insert fails (relying on leader consensus and state clearance to null on pod crash).
- Optimized 2-Bit Manipulation: Implementing a dedicated library or refined bitwise shifting logic for native 2-bit array manipulation.

1. FRONTEND (React.js)
--------------------------------------------------------------------------------
- Location: Get user location to fetch relevant city events.
- Home Page (Sliders): 'Trending' and 'Happening Soon' sections. Rectangular tiles containing Thumbnail, Event Name, and Date & Time subheading.
- Details Page: Extensive filtering options. Cards displaying all fetched event details.
- Search Bar: Backend-driven search with a 300ms debounce to prevent API flooding.
- Interactive Seat Map: 
  - Connects to BoSS WebSocket upon opening.
  - UI Legend (2-bit translation):
    * Light Grey (0): Empty / Available.
    * Orange (Animated sweep): Processing (Requested to lock).
    * Green (1 + UserID matched): Reserved by 'this' user.
    * Light Grey with Lock Icon (1): Reserved by someone else.
    * Dark Grey (2): Permanently Booked.
  - Interaction: Clicking an empty seat implicitly fires a lock request via WS.

--------------------------------------------------------------------------------
2. BACKEND: MAIN SERVICE (Node.js / Express.js)
--------------------------------------------------------------------------------
Auth:
- POST /register, POST /login. 
- Issues JWT with 2-hour expiration.
- DB Store: {id, name, email, location, address, username, mobile, hashed_password}.

Event Management:
- GET /events/{location}: Fetches events for a specific city. 
  - Table 'events': {id, code, name, description, artists, date_time, venue, location, total_seats, available_seats, bannerurl, thumbnailurl}.
  - Note: 'code' acts as a unique username-style slug for easy sharing.
- GET /search: Accepts static filters (location, has_artist, code, date/time window) and text_query. Search acts on the entire global space unless filtered.

Booking & Checkout (Standard HTTP REST):
- POST /events/{eventid}/checkout: Receives seat IDs currently locked by user.
  - Initiates Payment Gateway flow.
- POST /webhooks/payment: (Payment Confirmation)
  - Atomic Lua script to verify lock ownership and clear `<eventid>:lock:<seat_num>`.
  - Updates `<eventid>:status` Bitmap to '2' (Booked).
  - Inserts rows into 'seats_booked' and 'bookings'.
  - Publishes {seat_num, new_value: 2} to Redis PubSub `<eventid>`.
- DELETE /events/{eventid}/cancel: 
  - Cancels booking via HTTP. Updates 'seats_booked'/'bookings' in DB.
  - Updates Redis Bitmap back to '0' and publishes to PubSub.

--------------------------------------------------------------------------------
3. BACKEND: BoSS SERVICE (Go - Booking + Server Streaming)
--------------------------------------------------------------------------------
Architecture: One central Broadcaster Goroutine per active Event, managing multiple user WS connections.

WebSocket Connection Flow (/stream/{eventid}):
  i.   Auth Check.
  ii.  Fetch `<eventid>:status` Bitmap from Redis.
       -> [MVP NULL HANDLING]: If key doesn't exist (null), simply initialize the Redis string with an all '0' state (assuming an empty event for the MVP scope) and serve.
  iii. Send initial Bitmap state to user.
  iv.  Subscribe connection to Redis PubSub channel `<eventid>`.
  v.   On PubSub message -> Push JSON update to user: {seat_num, new_status: 0,1,2}.

WebSocket Client Messages (Locking/Unlocking):
  - Expected JSON: {reqid, seat_num: [], action: "LOCK" | "UNLOCK"}
  
  Case 1: LOCK
    a. Execute atomic Lua script: Check Bitmap. If '0', set to '1'. Also SET `<eventid>:lock:<seat_num>` with LOCK_TTL.
    b. Respond to user: {reqid, success: partial/true/false, failed_seats: []}.
    c. If successful, publish lock to PubSub `<eventid>`: {seat_num: [], new_status: 1}.
    d. Store locked seat IDs locally in the connection's memory state.

  Case 2: UNLOCK (Manual via WS)
    a. Verify ownership. If valid, update Bitmap to '0' and DEL `<eventid>:lock:<seat_num>`.
    b. Publish unlock to PubSub `<eventid>`: {seat_num: [], new_status: 0}.

  On WS Disconnect:
    a. Retrieve locally stored active locks for this connection.
    b. Execute UNLOCK logic for those specific seats (prevents waiting for TTL).

The Reconciler (Event-Level Goroutine):
  - Runs once every 30 seconds PER EVENT (not per user).
  - Compares Bitmap against actual active lock keys in Redis.
  - Fixes any discrepancies (Ghost Locks) caused by lost PubSub messages.
  - Pushes the reconciled Bitmap state to all users in the event room.

--------------------------------------------------------------------------------
4. BACKEND: UNLOCKER SERVICE (Go)
--------------------------------------------------------------------------------
Role: Dedicated consumer for Redis Keyspace Notifications (TTL Expirations).

Flow:
  i.   Consumes expired key event: `<eventid>:lock:<seat_num>`.
  ii.  Updates Bitmap at `<eventid>:status` to '0' (Empty) via native bitfield operation.
  iii. Publishes unlock to PubSub channel `<eventid>`: {seat_num, new_status: 0}.

--------------------------------------------------------------------------------
5. REDIS SCHEMA
--------------------------------------------------------------------------------
1. <eventid>:status       -> BitMap (2 bits per seat. 00=Empty, 01=Reserved, 10=Booked).
2. <eventid>:lock:<seat>  -> Ephemeral String (Value: UserID/SocketID, EX: LOCK_TTL).
3. PubSub Channel         -> Name: <eventid> (Broadcasts state deltas).
4. Keyspace Notifications -> Emits events when <eventid>:lock:<seat> expires.
